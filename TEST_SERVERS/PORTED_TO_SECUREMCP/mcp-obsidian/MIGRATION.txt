mcp-obsidian: low-level Server -> SecureMCP port
=================================================

WHAT WE DID (in plain English first)
------------------------------------

The original mcp-obsidian server used a dict-of-handlers pattern:

  - A `tool_handlers` dict mapped each tool name to a ToolHandler
    instance.
  - 13 ToolHandler subclasses lived in tools.py. Each subclass
    carried its own `name`, a `get_tool_description()` method that
    returned a Tool() with literal inputSchema, and a `run_tool(args)`
    method that did the actual work (called the Obsidian REST API).
  - server.py wired this up with @app.list_tools (loop over the
    dict) and @app.call_tool (look up by name, call run_tool).

Under SecureMCP this dispatch pattern is replaced by 13 individual
@app.tool() functions, one per tool. The 13 ToolHandler subclasses
in tools.py are NOT changed -- the port keeps a single instance of
each at module level and the wrapper functions delegate to
handler.run_tool(args).

Why we kept tools.py untouched: the inputSchema literals in
get_tool_description are useful documentation reference, the
run_tool bodies are the actual Obsidian-API-calling code, and not
modifying that file means the bulk of the original logic stays
verifiable against git history.

WHAT CHANGED, FILE BY FILE
--------------------------

src/mcp_obsidian/server.py -- rewritten:

  1. Import swap:
       from mcp.server import Server
       from mcp.types import Tool, TextContent, ImageContent, EmbeddedResource
     ->
       from macaw_adapters.mcp import SecureMCP
       (Tool / TextContent / ImageContent / EmbeddedResource imports
        commented -- no longer needed in server.py since the wrappers
        return plain strings.)

  2. Constructor swap:
       app = Server("mcp-obsidian")
     ->
       app = SecureMCP("mcp-obsidian")

  3. tool_handlers dict + add_tool_handler() + get_tool_handler()
     dropped. The dict + helpers were SecureMCP's job under the new
     pattern; we register tools via @app.tool() decorators instead.
     Original code preserved as a comment block.

  4. add_tool_handler(...) calls (one per ToolHandler) replaced by
     module-level instances:
       _LIST_VAULT = tools.ListFilesInVaultToolHandler()
       _LIST_DIR = tools.ListFilesInDirToolHandler()
       ...
     (13 instances total). Same instantiation semantic as the
     original, just stored as named globals instead of dict values.

  5. @app.list_tools / @app.call_tool dispatcher commented out.
     Replaced by 13 @app.tool() wrappers.

  6. _extract_text() helper added (~5 lines). Pulls .text out of a
     handler's [TextContent(...)] return so the wrappers can return
     plain strings -- avoids the awkward `{"result": [TextContent...]}`
     wrap on the mesh side.

  7. 13 @app.tool() wrappers added (~150 lines). Each wrapper:
        - has typed Python kwargs matching the original inputSchema
        - uses @app.tool(name="obsidian_<name>") to preserve the
          original advertised tool name byte-for-byte
        - calls handler.run_tool({...}) (sync method)
        - returns _extract_text(result) as a plain string
     Note on wrapper 11 (obsidian_get_periodic_note): the original
     parameter name is `type` (which shadows the Python built-in
     inside the function scope). We kept it as `type` because the
     JSON-schema parameter name is what existing callers pass --
     renaming it would break the integration. Harmless inside the
     function.

  8. async main() with stdio_server replaced by sync main() with
     app.run(). Same pattern as every other port in this series.
     Original code preserved as a comment block.

src/mcp_obsidian/__init__.py -- 1 edit:
  asyncio.run(server.main()) -> server.main()
  server.main() is sync now; wrapping a sync return-None call in
  asyncio.run() raises TypeError. Both the import and the wrapped
  call are preserved as comments.

src/mcp_obsidian/tools.py -- UNTOUCHED.
  All 13 ToolHandler subclasses remain byte-for-byte. The
  inputSchema literals serve as documentation reference. The
  run_tool bodies are still the active code path -- called by
  every wrapper.

src/mcp_obsidian/obsidian.py -- UNTOUCHED.
  The Obsidian REST client. No MCP imports. Pure business logic.

WHY (one line per change)
-------------------------

1. SecureMCP is the MACAW-instrumented MCP server class. The
   import is the only thing connecting this server to MACAW.
2. Same reason -- constructor must reference the new class.
3. tool_handlers dict + helpers were SecureMCP's job; @app.tool()
   replaces both the registration and the lookup.
4. Module-level instances match the original add_tool_handler
   semantic (one ToolHandler instance per tool, persistent for
   the process).
5. @app.tool() decoration takes over what list_tools / call_tool
   did: advertise the name + dispatch to the handler.
6. _extract_text avoids returning [TextContent(...)] which would
   wrap awkwardly on the mesh.
7. The wrappers preserve every tool name (via @app.tool(name=...))
   and every inputSchema field (as typed Python kwargs). Same
   public API.
8. SecureMCP.run() is sync; the original async main + stdio_server
   would conflict with SecureMCP's internal asyncio.run().

BROKEN ON PURPOSE -- features intentionally lost
------------------------------------------------

(a) Per-parameter inputSchema descriptions.

    Original: each parameter had a `description` field in the
    Tool() inputSchema, visible to MCP clients (e.g. for query
    in obsidian_simple_search: "Text to a simple search for in
    the vault.").

    Port: descriptions have moved into each wrapper function's
    docstring `Args:` block. SecureMCP auto-uses the docstring
    as the tool description (mcp.py:499). LLM clients still see
    the info, just as prose rather than structured per-property
    JSON. Same pattern as spotify / k8s ports.

    Restoring this would require SDK changes (out of scope).

(b) Tool result envelope shape.

    Original: handlers returned `Sequence[TextContent | ...]` --
    the low-level Server result envelope.

    Port: wrappers extract .text and return plain strings.
    SecureMCP wraps non-dict returns as `{"result": <str>}`
    (mcp.py:815-817). Mesh consumers see the same content under
    a `result` key. Same pattern documented in every prior port.

(c) ImageContent / EmbeddedResource return types.

    Original: handlers' return type was
    `Sequence[TextContent | ImageContent | EmbeddedResource]`.

    Port: only the .text field is extracted. None of the existing
    13 ToolHandler subclasses actually return ImageContent or
    EmbeddedResource -- every run_tool body returns
    [TextContent(...)] -- so this is a documentation difference,
    not a functional loss. If a future tool needs to return an
    image, the wrapper will need a small adjustment to base64-
    encode and return as a dict (skill Step 2b pattern).

VERDICT: PORT SUCCESSFUL (gold-standard, with full server-log
evidence).

  Server-side logs confirm every layer ran exactly as designed.
  The full stack trace from the OBSIDIAN_API_KEY=fake run shows:

    macaw_adapters/mcp/mcp.py:807 (handler dispatch)
      ↓ result = func(**params)
    server.py:165 (port wrapper obsidian_list_files_in_vault)
      ↓ _extract_text(_LIST_VAULT.run_tool({}))
    tools.py:49 (unchanged ToolHandler.run_tool)
      ↓ api.list_files_in_vault()
    obsidian.py:56 (unchanged business logic)
      ↓ requests.get("https://127.0.0.1:27124/vault/", ...)
    Connection refused (no Obsidian REST API listening)

  Every line of port code traversed cleanly. The only failure
  is at the network layer: no Obsidian REST API plugin on
  127.0.0.1:27124 to answer the request. The mesh -> wrapper
  -> ToolHandler -> Obsidian client chain is verifiably
  intact.

  TEST 3 additionally proves the typed-kwarg pathway: for
  obsidian_simple_search(query='macaw', context_length=50)
  the URL constructed at the bottom of the stack was
  `/search/simple/?query=macaw&contextLength=50` -- both
  kwargs survived the typed-kwarg -> dict-rebuild -> ToolHandler
  unpack -> URL-param translation. The 13-class -> 13-wrapper
  collapse + the kwarg-to-dict rebuild are both verified
  working with real call data.

  MACAW's response handling is correct too: when the handler
  raised the network exception, MACAW caught it, denied the
  call with policy=False (because the handler errored), and
  audit-logged the underlying reason:

    AUDIT: Service tool execution DENIED |
      Tool: obsidian_list_files_in_vault |
      Reason: Request failed: HTTPSConnectionPool(...)
              port=27124 ... Connection refused

  The earlier client-side "No result received for request" was
  the audit-denied response surfacing to the smoke client.
  That is the security-correct path: when the handler can't
  fulfil the request, deny the call and log the reason.

  Remaining work outside the port:
    - Real Obsidian verification with a live REST API plugin
      on the configured OBSIDIAN_HOST will produce real file
      listings and search results rather than denied calls.
      Validates Obsidian access, not the port.
    - The api.macawsecurity.ai audit-event timeout warnings
      are MACAW backend noise -- transient, not a port issue.

End-to-end runtime verification (real file CRUD, real searches
on a live vault) requires an Obsidian REST API plugin running on
the configured OBSIDIAN_HOST with a valid OBSIDIAN_API_KEY -- out
of scope for this smoke test.

KNOWN GAPS
----------

- This port has not been runtime-tested against a live Obsidian
  instance. Boot path is structurally correct; mesh-layer
  verification with real vault data needs the REST API plugin
  reachable.

- Per-parameter descriptions live in docstrings now, not in
  structured JSON. See BROKEN ON PURPOSE (a).

- ImageContent / EmbeddedResource paths in the return-type
  signature are documentation-only at present. See BROKEN ON
  PURPOSE (c). No current tool exercises them.
