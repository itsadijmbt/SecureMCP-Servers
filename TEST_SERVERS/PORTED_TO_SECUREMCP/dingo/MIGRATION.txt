dingo (Dingo Evaluator MCP Server): FastMCP -> SecureMCP port
==============================================================

WHAT WE DID (in plain English)
------------------------------

dingo is a tool-server for the Dingo data-quality evaluator. It's
a single 1318-line `mcp_server.py` script (top-level, not a
package) with 6 @mcp.tool() functions. The original used the
third-party `fastmcp` package (gofastmcp.com), but only its
absolute-minimum API surface: FastMCP("name") + @mcp.tool().

The port is mechanical:
  - Swap `from fastmcp import FastMCP` to
    `from macaw_adapters.mcp import SecureMCP`.
  - Swap the constructor.
  - Drop the multi-transport branching at the bottom (the original
    picked stdio vs SSE based on LOCAL_DEPLOYMENT_MODE; under
    SecureMCP both converge to plain mcp.run() because the mesh
    carries traffic regardless).

No tool function bodies touched, no Field defaults to rewrite,
no ctx.<log> awaits to strip, no lifespan, no _mcp_server access,
no custom_route, no decorator kwargs.

WHY (one line per change)
-------------------------

1. SecureMCP is the MACAW-instrumented MCP server class. Even though
   dingo used the third-party fastmcp package (gofastmcp.com), the
   API surface dingo actually exercised is the FastMCP("name") +
   @mcp.tool() core -- which SecureMCP supports identically. The
   import swap is mechanical.

2. Same reason -- constructor must reference the new class.

3. Transport selection is inert under SecureMCP -- mcp.run() ignores
   the `transport=` kwarg (mcp.py:628-635) and creates its own
   asyncio.run() internally. Traffic arrives via the MACAW mesh,
   not stdio or SSE. The LOCAL_DEPLOYMENT_MODE branch is collapsed
   because both paths converge to identical SecureMCP behaviour.

BROKEN ON PURPOSE -- features intentionally lost
------------------------------------------------

(a) SSE transport mode.

    Original: when LOCAL_DEPLOYMENT_MODE was unset (default), the
    server ran in SSE mode at fastmcp's default host/port,
    publishing the MCP-protocol stream over HTTP+SSE. MCP clients
    like Claude Desktop / Smithery could connect to it.

    Port: SecureMCP has no SSE transport. The MACAW mesh carries
    traffic instead. Existing SSE clients won't connect to this
    SecureMCP-ported server -- they'd need to migrate to the mesh.
    For deployments that must keep SSE, the original mcp_server.py
    can run alongside SecureMCPProxy (skill Step 7 -- "alongside
    uvicorn is allowed").

(b) Tool result envelope shape.

    Original: tools returned plain `str` and FastMCP wrapped them
    in MCP TextContent.

    Port: SecureMCP wraps non-dict returns as `{"result": <str>}`
    (mcp.py:815-817). Mesh consumers see the same string content
    inside a `result` key. Same pattern documented in every prior
    port.

(c) Future fastmcp-third-party features.

    NOTE: dingo today uses ONLY the FastMCP("name") + @mcp.tool()
    core. If a future maintainer adds third-party-fastmcp-specific
    features (Context, OAuthProxy, custom_route, lifespan kwarg
    on the FastMCP constructor), the import swap will break and
    will need to be revisited. This is documented here so the
    "minimal surface" assumption is on record.

KNOWN GAPS
----------

- This port has not been runtime-tested against a live dingo
  evaluation. Boot path is structurally correct; mesh-layer
  verification with real data needs a configured input file.

- SSE transport mode is gone. See BROKEN ON PURPOSE (a).

- Per-parameter inputSchema descriptions: dingo's original tools
  carry their docs in function docstrings (Args: blocks), not
  Pydantic Field metadata. Nothing structural lost.

NOTE on the third-party fastmcp package
---------------------------------------

dingo's `from fastmcp import FastMCP` was the gofastmcp.com
package, NOT `mcp.server.fastmcp` from the official MCP SDK.
The two packages have overlapping but not identical APIs. SecureMCP
(at `secureAI/macaw_adapters/mcp/mcp.py`) is a subset of the
OFFICIAL FastMCP, not the third-party one.

Why the port works anyway: dingo's actual usage of `fastmcp` is
the absolute minimum -- FastMCP class instantiation + @mcp.tool()
decoration. Both APIs share that core verbatim. We did not need
to translate any third-party-specific feature. If a future change
introduces Context, OAuthProxy, custom_route, or lifespan kwargs
to dingo, the import swap must be revisited because those features
are NOT in SecureMCP.

This is the same situation we documented for magg (where the
third-party-fastmcp surface was substantial -- magg PROXY'd) vs.
dingo (where the surface was minimal -- dingo PORTed).
