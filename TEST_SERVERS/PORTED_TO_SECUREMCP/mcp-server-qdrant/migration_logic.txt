================================================================================
MIGRATION LOG: mcp-server-qdrant  →  SecureMCP
================================================================================

Goal: take the upstream qdrant MCP server (which is built on FastMCP) and
re-wire it so its two tools (qdrant-find, qdrant-store) run through MACAW's
security layer instead of FastMCP's plain HTTP/stdio transport.

Hard rule we followed throughout:
  Do NOT modify SecureMCP (secureAI/macaw_adapters/mcp/mcp.py).
  SecureMCP is shared baseline across postgres, falcon, qdrant, and future
  servers. Every change lives inside mcp-server-qdrant itself.


--------------------------------------------------------------------------------
1. WHAT WE CHANGED (file by file)
--------------------------------------------------------------------------------

File: src/mcp_server_qdrant/mcp_server.py

  a) Import swap
       before: from mcp.server.fastmcp import Context, FastMCP
       after : from macaw_adapters.mcp import Context, SecureMCP

     The Context type looks the same to the tool functions, but it now comes
     from MACAW and carries identity / audit / policy hooks internally.

  b) Base class swap
       before: class QdrantMCPServer(FastMCP):
       after : class QdrantMCPServer(SecureMCP):

     This single line is the "integration." By inheriting from SecureMCP,
     the server registers itself as a MACAW agent on startup and every
     tool it declares goes through MACAW's invoke_tool pipeline.

  c) __init__ rewrite
     - Added a `name` parameter with default "securemcp-server-qdrant".
     - Dropped the FastMCP-only `instructions` param (not used by SecureMCP).
     - Preserved all qdrant-specific args (tool_settings, qdrant_settings,
       embedding_provider_settings, embedding_provider) because the rest of
       qdrant's code still uses them.
     - Called super().__init__(name=name, **kwargs) exactly ONCE at the end,
       after building self.qdrant_connector. (Early drafts had a duplicate
       super() call + an undefined `**settings` expansion — that was a bug;
       we removed it.)

  d) Tool registration idiom
     FastMCP supports the dual-call shape:
         self.tool(fn, name="qdrant-find", description=...)
     SecureMCP only supports the decorator shape:
         self.tool(name="qdrant-find", description=...)(fn)

     We changed both registrations to the decorator-application form:

         self.tool(
             name="qdrant-find",
             description=self.tool_settings.tool_find_description,
         )(find_foo)

         if not self.qdrant_settings.read_only:
             self.tool(
                 name="qdrant-store",
                 description=self.tool_settings.tool_store_description,
             )(store_foo)

     IMPORTANT: we did not change SecureMCP to accept the dual-call shape.
     We adapted qdrant's call sites to match SecureMCP's idiom — that is
     the rule.

  e) Logging calls (ctx.debug)
     FastMCP's Context.debug is `async` (you had to `await` it).
     SecureMCP's Context.debug is SYNC.
     We removed the `await` in front of every ctx.debug call inside
     `store()` and `find()`. Leaving an `await` would raise
     "object NoneType can't be used in 'await' expression" at runtime.

  f) QdrantConnector stays a plain instance attribute
     The connector is process-scoped (one shared instance for all callers).
     We store it on `self.qdrant_connector` inside __init__. We do NOT put
     it in MACAW's _session_objects dict — that dict is per-session state,
     not process state. Qdrant doesn't have a per-user session model; one
     connector per process is correct.


File: src/mcp_server_qdrant/main.py

  Removed the argparse block and the `--transport` CLI flag.
  main() now does:
       from mcp_server_qdrant.server import mcp
       mcp.run()

  Why: with MACAW, transport is cosmetic. Once the agent is registered on
  the mesh, callers reach tools via invoke_tool — nobody dials the
  FastMCP HTTP/stdio port. The run(...) call exists only to keep the
  process alive so the registered agent stays reachable.


File: src/mcp_server_qdrant/server.py

  Untouched. It still does:
       mcp = QdrantMCPServer(tool_settings=..., qdrant_settings=...,
                             embedding_provider_settings=...)
  Because QdrantMCPServer now inherits SecureMCP, instantiation is
  enough — registration with the MACAW mesh happens inside
  SecureMCP.__init__.



--------------------------------------------------------------------------------
2. WHAT WE PROVED WITH TESTING
--------------------------------------------------------------------------------

Server startup log showed:
    Registered as agent: local:itsadijmbt/app:securemcp-...-qdrant:<hash>
    Tools: ['qdrant-find', 'qdrant-store']

That alone proved:
  - imports resolve
  - __init__ ordering works (no pydantic errors)
  - QdrantConnector constructed
  - both tools registered via SecureMCP's decorator
  - register() succeeded on the mesh

Client round-trip (TEST 3: qdrant-find "why is the sky blue") returned:
    {'result': [
        "Results for the query 'why is the sky blue'",
        '<entry><content>The sky is blue because of Rayleigh scattering.'
         '</content><metadata>...</metadata></entry>',
        ...
    ]}

That proved:
  - store → embed → vector search → find works end-to-end
  - Annotated[Metadata | None, Field(...)] survived SecureMCP schema generation
  - ctx.debug(...) (sync) works inside handlers
  - MACAW audit entries fire on both success and failure (we saw a
    "DENIED" audit when collection_name was missing — a real policy signal)


--------------------------------------------------------------------------------
3. PITFALLS WE HIT AND HOW TO AVOID THEM
--------------------------------------------------------------------------------

- QDRANT_LOCAL_PATH + async = process-level lock contention → tool calls
  time out after the first one. Fix: run qdrant in docker and use
  QDRANT_URL=http://localhost:6333 instead.


- The client's tool return is {'result': ...} — always unwrap before
  asserting on content.

- The agent name gets the "securemcp-" prefix twice if you pass
  name="securemcp-..." to __init__. Use a base name like "server-qdrant"
  so the final ID reads "securemcp-server-qdrant".


--------------------------------------------------------------------------------
4. WHAT WE DID NOT DO (deferred)
--------------------------------------------------------------------------------
- Docker qdrant sidesteps this because the lock now lives inside the qdrant container, on a different process with its own threads. Your Python event loop is free to do other things while
    waiting for an HTTP response earlier i.e QDRANT_LOCAL_PATH tells qdrant to run embedded — it's not a separate server process

- No changes to QdrantConnector, embedding providers, filter logic, or
  any business code. Only the MCP surface was touched.

- SecureMCP SDK was not modified. All integration work lives inside
  mcp-server-qdrant/. This is the baseline-stability rule.
