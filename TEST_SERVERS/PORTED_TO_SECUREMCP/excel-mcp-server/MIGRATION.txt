excel-mcp-server: FastMCP -> SecureMCP port
============================================

WHAT WE DID (in plain English first)
------------------------------------

The original excel-mcp-server is a clean FastMCP server with 25 tools
(read/write Excel files via openpyxl). Every tool is decorated with
`@mcp.tool(annotations=ToolAnnotations(title=..., readOnlyHint or
destructiveHint=True))`. Three thin wrapper functions (`run_sse`,
`run_streamable_http`, `run_stdio`) each call `mcp.run(transport="...")`.
A `__main__.py` exposes those three as typer subcommands.

This is the same shape as k8s-mcp-server -- pure FastMCP, decorator-
kwargs to strip, multi-transport main to neutralize. Mechanical port,
no business logic touched, no architectural decisions to make beyond
the ones already documented in the skill.

WHAT CHANGED, FILE BY FILE
--------------------------

src/excel_mcp/server.py -- edited (866 -> 955 lines, +89 for
preserved-as-comment originals + PORT NOTE blocks):

  1. Import swap (lines 5-6):
       from mcp.server.fastmcp import FastMCP
       from mcp.types import ToolAnnotations
     ->
       from macaw_adapters.mcp import SecureMCP
       (ToolAnnotations import commented out -- only used for the
        annotations= kwarg we strip in step 4 below.)

  2. Constructor swap (lines 68-73):
       mcp = FastMCP(
           "excel-mcp",
           host=os.environ.get("FASTMCP_HOST", "0.0.0.0"),
           port=int(os.environ.get("FASTMCP_PORT", "8017")),
           instructions="Excel MCP Server for manipulating Excel files"
       )
     ->
       mcp = SecureMCP("excel-mcp")
     SecureMCP doesn't accept host=, port=, or instructions= (skill
     Step 1b). The FASTMCP_HOST / FASTMCP_PORT env vars become unused
     under this port -- noted in the in-code PORT NOTE for anyone
     migrating an existing deployment.

  3. ctx async-to-sync (Step 5): N/A -- excel-mcp-server doesn't use
     ctx.<log> methods anywhere.

  4. Decorator kwarg strip (25 instances). Each
       @mcp.tool(
           annotations=ToolAnnotations(
               title="...",
               readOnlyHint or destructiveHint=True,
           ),
       )
     became
       @mcp.tool()
     Originals preserved as comments inline above each stripped
     decorator. Skill Step 1c. Scripted (same approach used in
     telegram-mcp / k8s-mcp-server). 25 patches verified.

  5. Field-default rewrite (Step 2): N/A -- excel-mcp-server uses
     plain Python defaults for all parameters; no Field(...) on
     parameter lines.

  6. Transport functions at the bottom (run_sse / run_streamable_http
     / run_stdio): each had `mcp.run(transport="...")` replaced with
     `mcp.run()`. Originals preserved as comments inline. The three
     functions remain as separate Python functions because
     __main__.py exposes them as three typer subcommands; under the
     port all three boot the same SecureMCP server, but the user-
     facing CLI (`python -m excel_mcp <subcommand>`) is unchanged.

src/excel_mcp/__main__.py -- UNCHANGED.

  The typer wrapper just dispatches to run_sse / run_streamable_http
  / run_stdio. Those functions still exist with their original
  signatures; only their bodies' `mcp.run(transport=...)` lines
  changed to `mcp.run()`. The CLI surface is preserved.

BROKEN ON PURPOSE -- features intentionally lost
------------------------------------------------

(a) Tool advertisement annotations.

    Original: each @mcp.tool carried
    annotations=ToolAnnotations(title="Apply Formula",
    destructiveHint=True), etc. MCP-protocol clients used these
    hints in their UI (display title, warn before destructive
    operations like write_data_to_excel).

    Port: SecureMCP @mcp.tool() does not accept annotations=.
    Stripped mechanically. Tool function names + docstrings are
    preserved -- mesh consumers can read those and infer the same
    information.

    Restoring this would require SDK changes (out of scope).

(b) Tool result envelope shape.

    Original: tools returned plain Python strings; FastMCP wrapped
    them in MCP TextContent.

    Port: SecureMCP wraps non-dict returns as `{"result": <str>}`
    (mcp.py:815-817). Mesh consumers see the same string content
    inside a `result` key. Same pattern documented in spotify /
    telegram / k8s / azure-devops / mysql ports.

KNOWN GAPS
----------

- This port has not been runtime-tested with real Excel files.
  Boot path is structurally correct; mesh-layer verification with
  real openpyxl-touched files needs the smoke test run against a
  prepared EXCEL_FILES_PATH.

- Tool annotations (titles, readOnlyHint, destructiveHint) are
  lost. UI clients that warned before destructive writes (e.g.
  write_data_to_excel, delete_worksheet) no longer get that hint
  surfaced. See BROKEN ON PURPOSE (a).

- HTTP transport bind addresses (FASTMCP_HOST / FASTMCP_PORT) are
  unused. See BROKEN ON PURPOSE (c).
