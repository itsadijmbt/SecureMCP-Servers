arxiv-mcp-server: low-level Server -> SecureMCP port
======================================================

WHAT WE DID (in plain English first)
------------------------------------

The original arxiv-mcp-server used the low-level MCP Server pattern:

  - One @server.list_tools() that returned 10 hand-built Tool() objects.
  - One @server.call_tool() with `if/elif name == "search_papers": ...`
    branching to 10 handler functions.
  - A prompts subsystem (4 prompts: paper analysis, summarize paper,
    compare papers, literature review) wired via @server.list_prompts
    and @server.get_prompt.
  - Transport machinery for two modes: stdio and streamable-HTTP
    (uvicorn + Starlette + StreamableHTTPSessionManager + DNS
    rebinding protection).

Under SecureMCP the dispatcher pattern is gone -- each tool becomes
its own @app.tool() function. The prompts subsystem is dropped. The
transport machinery is also dropped (mesh carries traffic; there is
no MCP-stdio listener, no HTTP listener).

The 10 handler functions in tools/<name>.py are NOT changed. The
port wraps each of them in a thin @app.tool() function that takes
typed kwargs, rebuilds the dict the handler expects, and forwards
to the unchanged handler. Same data, same code path, just a clean
Python signature where the original had a generic `arguments: dict`.

ROOTS DECISION -- why we dropped the prompts subsystem
------------------------------------------------------

The original prompts subsystem exposed 4 prompts (paper analysis,
summarize paper, compare papers, literature review) through
@server.list_prompts and @server.get_prompt. The get_prompt handler
took session-aware arguments (expertise_level, paper_id, ...) and
returned a constructed GetPromptResult.

Three options were considered:

  PATH A (CHOSEN, active in this port) -- DROP prompts entirely.
    @server.list_prompts and @server.get_prompt are commented out.
    The 4 prompt files in prompts/ remain on disk, untouched. They
    are simply no longer registered with SecureMCP.

  PATH B (rejected) -- convert each prompt to a @app.prompt()
    decorator. Would preserve "the prompt is available" but lose
    the session-aware argument handling (SecureMCP's @app.prompt()
    is a static handler -- no session args). Medium diff. Requires
    reading each prompt file's content and porting to a function.

  PATH C (rejected) -- expose each prompt as a @app.tool().
    Largest behavioural shift -- changes the integration pattern
    from prompts/get to tools/call.

Why we chose Path A:
  - Prompts are nice-to-have for arxiv-mcp-server. The core
    functionality (search, download, read papers) is in the 10
    tools, all of which port cleanly.
  - The session-aware argument handling has no clean SecureMCP
    equivalent. Trying to preserve it would require either
    modifying SecureMCP (forbidden by the project rule) or
    building a tool-based imitation that changes the client-side
    integration anyway.
  - Path A is reversible. The 4 prompt files are still on disk,
    so a future Path-B implementation can pull them in via
    @app.prompt() decorators without disturbing the tool ports.
  - This is the kind of decision Hard Rule #11 was written for,
    and it was raised explicitly to the user before any code was
    written.

BROKEN ON PURPOSE -- features intentionally lost
------------------------------------------------

(a) Prompts subsystem.

    Original: 4 prompts available via MCP `prompts/list` and
    `prompts/get` with session-aware argument handling.

    Port: dropped. The prompt files remain on disk in prompts/
    but are not registered with SecureMCP. Mesh consumers who
    relied on these prompts will not see them via prompts/list.

    Restoration path: re-register the prompts via @app.prompt()
    decorators (Path B above). Each decorator would be a thin
    function returning the prompt's text content. Session-aware
    argument handling would not survive (no SecureMCP equivalent).

(b) Tool advertisement annotations.

    Original: a few tools (read_paper, list_papers, get_abstract,
    download_paper, search_papers) carried
    annotations=ToolAnnotations(readOnlyHint=True, ...) in their
    Tool() definition. MCP-protocol clients used those for UX.

    Port: SecureMCP @app.tool() does not accept annotations=. The
    Tool() definitions in tools/<name>.py are unused and unread by
    the port; their annotations come along for free as dead code.

    Restoration would require SDK changes (out of scope).

(c) Per-parameter inputSchema descriptions.

    Original: each parameter had a `description` field in the
    Tool() inputSchema, visible to MCP clients.

    Port: per-parameter descriptions have moved into each wrapper
    function's docstring `Args:` block. SecureMCP auto-uses the
    docstring as the tool description (mcp.py:499). LLM clients
    still see the info, just in prose form rather than structured
    per-property JSON. Same pattern as the spotify / k8s ports.

(d) HTTP / SSE / streamable-http transport.

    Original: TRANSPORT env var picked stdio or streamable-HTTP.
    The HTTP mode used uvicorn + Starlette + StreamableHTTPSession
    Manager with DNS rebinding protection.

    Port: SecureMCP carries traffic over the MACAW mesh. The HTTP
    listener is gone. ARXIV_HOST / ARXIV_PORT / ARXIV_TRANSPORT /
    ARXIV_ALLOWED_HOSTS / ARXIV_ALLOWED_ORIGINS env vars become
    unused.

(e) MCP `serverInfo` (server name + version) advertisement.

    Original: _initialization_options() built MCP server-info with
    settings.APP_NAME and settings.APP_VERSION.

    Port: dropped. SecureMCP exposes identity via agent_id; the
    APP_VERSION is still in settings for any consumer who reads
    it directly.

(f) Tool result envelope shape.

    Original: handlers returned `List[TextContent]` -- the
    low-level Server result envelope.

    Port: wrappers extract .text and return a plain string.
    SecureMCP wraps non-dict returns as `{"result": <str>}`
    (mcp.py:815-817). Same pattern as every other port in this
    series.

(g) Error-raising via _tool_error_message: PRESERVED.

    The original raised RuntimeError if a handler returned an
    error payload (json with "status": "error" or text starting
    with "Error:"). The port keeps this behaviour -- each
    wrapper calls _tool_error_message after the handler returns
    and raises if an error is detected. Mesh consumers see the
    same exception surface as the original.

KNOWN GAPS
----------

- This port has not been runtime-tested against arXiv. Boot path
  is structurally correct; mesh-layer verification with real
  arXiv calls needs network access and the smoke test running
  against a freshly booted server.

- Prompts are dropped. See ROOTS DECISION + BROKEN ON PURPOSE (a).

- Tool annotations / per-parameter inputSchema descriptions live
  in docstrings now, not in structured JSON. See BROKEN ON
  PURPOSE (b)/(c).

- HTTP transport is gone. See BROKEN ON PURPOSE (d).
