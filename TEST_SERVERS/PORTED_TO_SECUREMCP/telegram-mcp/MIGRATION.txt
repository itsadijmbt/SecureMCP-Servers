  5. Entry-point rewrite (lines 6160-6188):
     The original async _main() did three things before serving:
       (a) await cl.start() for every Telegram client (eager async
           connect, opens anyio scope on the loop);
       (b) await cl.get_dialogs() for every client (cache warming);
       (c) await mcp.run_stdio_async() (FastMCP's async stdio entry).
     And on shutdown:
       (d) await cl.disconnect() in finally.

     Under SecureMCP, mcp.run() is sync and creates its own
     asyncio.run() internally. (a)+(b) bound to a loop that would
     close before SecureMCP started -- the dead-reactor pattern.

     Replacement:
       (a) is dropped. The existing ensure_connected(cl) helper
           (line ~231 in original) already does lazy connect-on-
           first-call inside tool handlers, on SecureMCP's loop.
           Same security and same connection lifetime, slightly
           slower first call.
       (b) is dropped. Cache warming was a pure latency
           optimisation; first dialog lookup pays one extra
           Telegram round-trip, then telethon caches.
       (c) becomes mcp.run() (sync).
       (d) is dropped. SecureMCP exposes no async shutdown hook;
           OS reclaims sockets on process exit.
     nest_asyncio.apply() is dropped (no nested loops under
     SecureMCP). asyncio.run(_main()) becomes a plain sync main()
     body that calls mcp.run().

  Original async _main() and old main() preserved as a comment
  block immediately above the new sync main().

ROOTS PORT DECISION (the load-bearing call)
-------------------------------------------

The original FastMCP server gated file-path tools (upload_file,
download_media, ...) using `await ctx.session.list_roots()`. This
is the MCP-protocol "roots capability" -- the CLIENT declares its
allowed filesystem roots at session init, the server queries them
dynamically. SecureMCP has NO equivalent protocol slot
(verified at mcp.py:38-470: no Context.session.list_roots; only a
sync Context.get_roots() that returns SERVER-declared roots from
CLI args).

Three options were considered:

  PATH A (chosen, active in this port):
    Server-side roots only. _configure_allowed_roots_from_cli()
    parses --allowed-root CLI args at startup; every file-path
    tool reads SERVER_ALLOWED_ROOTS exclusively. The ctx parameter
    is kept on _get_effective_allowed_roots_with_status() for API
    stability but no longer consulted for roots.

  PATH B (preserved as commented alternative):
    Add a new `register_roots(ctx, roots)` tool that lets clients
    declare per-session roots; store via
    ctx.set_state("client_roots", ..., serializable=False) and
    read by the same helper. Falls back to SERVER_ALLOWED_ROOTS
    if no register_roots call this session.

  PATH C (rejected without serious analysis):
    Roots stored in MACAW vault keyed by user identity. Requires
    deployment-time vault wiring; overkill for telegram's typical
    single-tenant deployment.

WHAT PATH B IS, IN PLAIN TERMS
------------------------------

Path B preserves the per-session, client-supplied roots semantic
of the original by EXPLICITLY asking the client to call a tool
once at the start of its session:

    register_roots(roots=["/Users/me/Downloads", ...])

Roots are stashed in MACAW session state (ctx.set_state) and read
by the same helper that previously called ctx.session.list_roots().
Same security model (client decides; server gates by that list);
different choreography (explicit tool call instead of implicit
protocol handshake).

Trade-offs:
  + Preserves the per-session client-supplied roots semantic.
  + Same security model -- only the client's declared roots are
    honoured.
  + Falls back to SERVER_ALLOWED_ROOTS if client doesn't call
    register_roots, so file tools degrade gracefully.

  - Adds a client-side onboarding step: the client must call
    register_roots BEFORE any file-path tool will see its roots.
    Existing FastMCP clients that relied on session.list_roots()
    must be updated.
  - Adds one new tool (~30 lines including docstring) to the
    server.
  - Skill rule "do not change architecture" strictly favours A,
    because Path B introduces a new client-server contract step.

WHY WE CHOSE PATH A
-------------------

Three reasons, in order:

  1. Deployment shape. Telegram-mcp is typically run as a single
     user's Telegram bot serving one human's chats. Roots are
     usually a fixed download/upload directory set once. The
     dynamic-client-roots feature was inherited from MCP-protocol
     completeness, not from a specific telegram product
     requirement. We have no evidence of a deployment that
     varies roots per session.

  2. Smaller diff. Path A requires zero new tools and no
     client-side change. Path B adds one tool and forces every
     downstream client to learn one new step. Smaller change is
     safer for a port.

  3. Skill discipline. The "do not change architecture" rule
     covers integration-level protocols, not just security. A
     new "you must call register_roots first" step is a
     contract change visible to clients. Path A makes no such
     change.

When you would switch to Path B:
  * If a real telegram client surfaces that historically relied
    on session.list_roots() and you want behavioural parity.
  * If a multi-tenant deployment appears where each session
    needs different roots and admin cannot enumerate them
    upfront.

To activate Path B:
  * Uncomment the `register_roots` @mcp.tool block (currently
    in the comment section near the top of the tool group).
  * Inside _get_effective_allowed_roots_with_status, replace
    the active Path A body with the commented Path B body
    (both are present side-by-side).
  * Document the client-side onboarding requirement in your
    integration docs.

BROKEN ON PURPOSE -- features intentionally lost
------------------------------------------------

(a) Per-tool annotations metadata.

    Original: 109 @mcp.tool decorators carried
    annotations=ToolAnnotations(title="...", readOnlyHint=...,
    openWorldHint=..., destructiveHint=...). LLM clients used
    these hints for UX (display titles, warn before destructive
    calls).

    Port: SecureMCP @mcp.tool() does not accept annotations=
    (verified at mcp.py:497-517). Stripped mechanically. Titles
    and hints are gone; tool descriptions and docstrings remain.

    Restoring this requires SDK changes -- out of scope for any
    integration. Mesh consumers should rely on tool description /
    docstring instead of the hint flags.

(b) Dynamic client-supplied filesystem roots.

    See ROOTS PORT DECISION above. Replaced by Path A
    (server-side CLI roots only); Path B preserved as a
    commented alternative.

(c) Eager Telegram client startup at server start.

    Original: cl.start() and cl.get_dialogs() ran before
    mcp.run_stdio_async(), so first tool calls were warm.

    Port: lazy connect on first tool use via ensure_connected().
    First tool call pays one extra Telegram round-trip; same
    correctness, slightly slower cold start.

(d) Async cleanup on shutdown.

    Original: cl.disconnect() awaited in _main()'s finally.

    Port: dropped. SecureMCP has no async shutdown hook.
    Telegram sessions remain valid; the server-side session is
    timed out by Telegram if needed. OS reclaims local sockets
    on process exit.

(e) Tool result envelope shape.

    Original: tools returned `str` (often JSON via json.dumps).
    FastMCP wrapped these as MCP TextContent.

    Port: SecureMCP wraps non-dict returns as {"result": <str>}
    (mcp.py:815-817). Mesh consumers see the same string content
    inside a result key. Documented for callers that parse the
    response.

KNOWN GAPS
----------

- This port has not been runtime-tested against a real Telegram
  account. Boot path verified; mesh-layer verification requires
  real credentials.

- Tool annotations (titles, readOnlyHint, etc.) are lost. If a
  consumer relied on these, see BROKEN ON PURPOSE (a).

- Per-session client-supplied roots are unsupported under Path A.
  If a consumer relied on dynamic roots, switch to Path B per the
  instructions in ROOTS PORT DECISION.
